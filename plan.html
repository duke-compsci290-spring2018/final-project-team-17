Design


We plan on having a single page application containing all of our functionality.


Upon arrival to the site, users will be prompted to login/signup, after which they will either see their current progress or a blank dashboard with indicators to help them get started (ex. flashing arrows and buttons).


The primary feature, which is the relationship tracker, will be the center console of the page (probably taking up 70% of the center of the screen). Other features will be fitted as panels on either side of the center. Clicking on a relationship will highlight or bring to attention any relevant information on either side of the screen.

Functionality and Data

The steps that we will take to build our application are as follows:

  1. In building the application, first we will develop a database schema for our data.
  2. We will then build read/write functionality for our database.
  3. Next, we will create specific features to notify users to complete relationship boosting activities.
  4. We will integrate user data into these features to update the “relationship quality” metric and improve push notifications (by updating a backend calendar, perhaps)
  5. We will implement a quiz functionality that can be used to generate a “relationship quality” score
  6. We will incorporate a feature to track items lent between acquaintances and friends
  7. Then, we will implement all of these features in a beautiful UI.

Relationship Boosting

For relationship boosting, we will allow new users to add people to their dashboard who they want to improve their relationship with.

Each relationship that the user adds will have the following attributes:

  • Name
  • Associativity (mother, sister, close friend, etc.)
  • Initial rating of closeness (slider scale of 1-100 [1 being worst and 100 being best]), OR
  • User can take a quiz to help him/her determine closeness with the relative/friend
  • Based on the answers to the questions, we determine the initial rating for them

Inputs from user:

  • Allow user to add interactions with each relationship (e.g. “text” or “email”)
  • Allow user to rate interactions
  • Allow user to provide a desription of a recent interaction

Calendar

Users can sync their Google Calendar to allow our platform to select the best time that allows for ample social interaction with a friend or relative on the user's list.

Inputs would be:

  • Dates and times of availability, OR
  • Manually-selected dates and times for availability

Lending

Users can keep track of things/money that they've lent to someone or someone has lent to them.

Data for this would be stored as:

  • Whether or not item has been loaned to or lent
  • Name of person involved
  • Time since lending action
  • Approximate monetary value of lent object

Stack

  • Node.js - server that reads/writes data into database
  • Mongo.db - to store user data
  • Tensorflow.js - for machine learning algorithms
  • An undecided front-end framework (possibly Vue) - for binding data to UI in realtime, makes it easier to have a single-page website
  • Twilio - for text notifications (or maybe something simpler and free)
